# Methodology

## Research Goal

The purpose of this project is to investigate whether AI-guided fuzzing can find software bugs more effectively than traditional random fuzzing. To do this, two fuzzing approaches will be implemented and tested on the same target program.

## Target Program

For the initial experiment, a simple calculator program was selected as the target. The calculator accepts arithmetic expressions as input and evaluates them using Python's Abstract Syntax Tree (AST) functionality.

The calculator was chosen because it is small, easy to understand, and provides a controlled environment for testing different fuzzing techniques. It can also generate a variety of errors when given malformed or unexpected inputs.

## Random Fuzzer

A random fuzzer was developed as the baseline approach. The fuzzer generates random strings containing numbers, operators, parentheses, and alphabetic characters. These inputs are then supplied to the calculator program.

When an error occurs, the error message and corresponding input are recorded in a log file. This allows the behavior of the random fuzzer to be analyzed later.

## AI-Guided Fuzzer

An AI-guided fuzzing component was also developed. At the current stage of the project, the fuzzer uses a set of manually selected test inputs that represent edge cases, invalid expressions, and unusual arithmetic operations.

In future stages of the research, these manually selected inputs will be replaced by inputs generated with the assistance of a Large Language Model. The goal is to determine whether AI-generated inputs can discover faults more efficiently than randomly generated inputs.

## Data Collection

During each experiment, the following information is recorded:

* Input provided to the target program
* Program output
* Error messages generated by the program
* Number of successful executions
* Number of failed executions
* Execution time for each test case
* Exception type associated with each failure

The collected data is stored in structured log files for later analysis and comparison.

## Evaluation

The performance of the two fuzzing approaches will be compared using several measurements. These include the total number of inputs tested, the number of errors discovered, the number of successful executions, and the variety of unique error types identified.

Additional measurements include code coverage statistics collected during fuzzing experiments. Coverage measurements provide insight into how much of the target program is exercised by generated test inputs and will be used as an important comparison metric between random fuzzing and AI-guided fuzzing.

## Coverage and Logging Infrastructure

To support future experimentation, an automated execution and logging pipeline has been developed. Generated inputs are executed automatically and the outcome of each execution is recorded.

For each test case, the framework records:

* Input value
* Execution result
* Execution time
* Error type
* Error message

Coverage measurements are collected using the coverage.py tool. Coverage reports can be generated in both terminal and HTML formats, allowing detailed inspection of which portions of the target program were exercised during testing.

Experiment summaries are automatically generated and stored in structured JSON format to support later statistical analysis and comparison.

## Experimental Procedure

The random fuzzer and AI-guided fuzzer are executed separately against the calculator program. Each fuzzer generates a collection of test inputs and submits them to the target program.

The resulting outputs and errors are recorded and analyzed. The results from both approaches are then compared to evaluate their effectiveness in identifying faults within the target program.

## Expected Results

The expectation is that AI-guided fuzzing will produce more meaningful test inputs and reach interesting program behaviors more quickly than purely random input generation. However, the experiments will be used to determine whether this assumption is supported by the collected data.
